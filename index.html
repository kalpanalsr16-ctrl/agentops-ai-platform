Batch vs real-time
Real-time (stream)
Every request → anomaly detection → P0 trigger. Latency <100ms. Safety flags, latency SLA. Does not run full eval — too expensive per request.
Batch (scheduled)
Weekly eval runs (Sunday 2am) + daily regression checks. 10% traffic sample after 30-day full-log window.
Why not real-time eval?
Running 50 eval prompts per request costs ~$0.40/req and adds 3–4s latency. Batch costs ~$0.002/agent/week. Real-time catches severity, batch catches depth.

Lumen is not a monitoring dashboard. It is an operating system for AI agent reliability — it predicts, diagnoses, decides, fixes, and learns.

What Lumen is
One-line pitch
"The AI Reliability OS that predicts agent failure before it happens, explains exactly why, and makes sure the fix actually ships."
The gap it fills
Datadog tells you your server is slow. It cannot tell you your AI agent's answers are wrong.
OpenAI Evals runs tests. It doesn't connect those tests to production incidents or fix tracking.
Lumen closes the loop: detection → attribution → fix → verified improvement — with audit trail.
Hardest product decisions
Batch eval vs real-time — why batch wins
$0.002/agent/week vs $0.40/request. Real-time catches severity (P0s), batch catches depth (hallucination drift). The two are complementary, not substitutes. Real-time cannot run 50-prompt eval at 100ms response time.
Recall vs precision — why we optimise for recall
A false negative (bad agent ships) costs 10× a false positive (good agent delayed). 22% false positive rate is accepted. Developers who fill manifests completely see <5% false positives.
Automation vs human review — the boundary
Automate the 80% with deterministic pass/fail. Human review gates: attribution confidence <70%, all safety flags, all P0 delist confirmations. Automation without human gates creates false security.
Rollout strategy
PHASE 1 · Q3 2026
Detect + Diagnose
Observability + attribution to 1 high-volume category. Success: 80% of known incidents attributed within 4h. Baseline established for time-to-root-cause metric.
PHASE 2 · Q4 2026
Decide + Fix
Risk Simulator + Execution Tracking + Experiments panel. Risk Simulator gates all publish decisions. Success: rejection −15pp, developer response rate >75%, fix SLA met >80%.
PHASE 3 · Q1 2027
Learn + Scale
System Intelligence live. Loop 3 quarterly. Auto-resolve rate >40%. Success: rejection <15%, zero P0/quarter, time-to-root-cause <2h.
